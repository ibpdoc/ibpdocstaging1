Initial analysis

Adam identified the type of information to store, which includes member contact information and descriptions of the items to lend. The app should also make it easy to track when members borrow and return items.

He has a variety of of items to loan, including books, CDs, DVDs, and periodicals. Common characteristics of these items include: a title, authors/actors/artists, a publisher, and a publication date. He decides to store information about items to lend in a Title object.

To track lending, Adam decided that a Check Out object would be a good container for the name of the borrower, title borrowed, date borrowed, and date due. To keep things simple for the prototype, he assumed that every title is unique and that if a member borrows multiple titles at the same time, they return them all on the same day.

Next Adam mapped application requirements to built-in Platform functionality:

  • The Platform User object provides a way to create login accounts for developers and end-users and to manage permissions by role. The app will store information about library employees and members as User records.
  • When you create an object, Platform automatically adds a Record Name field:
    • For the Title object, Adam can use this field to store item titles, and just add the other fields he needs.
    • For the Check Out object, a unique identification number would be better for the Record Name and he would prefer it to be generated automatically. Platform allows you to use a template to automatically generate the value of the Record Name field. Adam decided to add an Auto-Number field that will create the values used by the Record Name template.
  • The information about borrower and title will be available in the User and Title object, so relationships between Check Out, User, and Title will tie the information together.
  • Platform triggers can include formulas to set the current date and calculate the due date from the current date. The only date field that librarians will need to set explicitly will be when a member returns borrowed items.
  • Relationships include cardinality, that is, how many objects on one side can be related to how many objects on the other side of the relationship. The library app should support the following relationships:
    • One member can check out titles multiple times, but each check out record should only be associated with one member.
    • One check out can have multiple titles, but a title should only be associated with one check out at a time. However, since the same title will be associated with multiple check out records over time, the cardinality between checkout and title will have to be many-to-many.

With this information, Adam was ready to start his app foundation. Use the Quick Create wizard to start the foundation for your tutorial app.
